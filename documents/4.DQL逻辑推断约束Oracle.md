# DQL 逻辑推断约束 Oracle

## 1. 目标

本模块的目标不是预先计算完整正确结果，而是从 DQL 与 schema 中静态推断出“实际执行结果必须满足的逻辑约束”，再将这些约束与数据库实际返回结果进行比较。

如果实际结果违反这些约束，则可以判定为潜在逻辑 bug。

核心原则：

- 推断对象是 DQL 的逻辑结果，不是 MySQL `cursor.description` 等 metadata 暴露行为。
- 不依赖执行计划，不依赖优化器选择。
- 不要求知道完整 expected result，只检查结果必须满足的性质。
- MySQL metadata 行为差异应放在对比层单独标注，不反向污染核心逻辑推断规则。

## 2. 可实现的核心逻辑推断约束

### 2.1 结果列数量约束

推断内容：

- `SELECT a, b` 的结果列数为 2。
- `SELECT *` 根据当前作用域 schema 展开。
- `SELECT t.*` 根据表或别名对应的列展开。
- CTE、子查询、派生表的列数由其内部 SELECT 推断。
- `UNION`、`UNION ALL`、`INTERSECT`、`EXCEPT` 两侧必须返回相同列数。

比较方案：

- 执行 DQL。
- 获取实际结果的列数。
- 比较 `inferred_column_count == actual_column_count`。

Bug 判定：

- 实际列数与逻辑推断列数不一致。
- set operation 两侧列数不一致但 DBMS 没有报错。
- `*` 展开后列数错误。

可检测问题：

- 投影展开错误。
- CTE 或派生表列映射错误。
- set operation 列对齐错误。
- schema 变更后元数据缓存污染。

### 2.2 结果列逻辑类型约束

推断内容：

- 基础列引用类型。
- 算术表达式类型。
- 字符串函数返回类型。
- 日期时间函数返回类型。
- 聚合函数返回类型。
- 条件函数类型合并，例如 `IF`、`CASE`、`COALESCE`、`NULLIF`。
- set operation 的隐式公共类型合并。
- binary/blob 参数对字符串函数返回类型的传播影响。

比较方案：

- 静态推断每个结果列的逻辑类型族，例如：
  - `numeric`
  - `string`
  - `datetime`
  - `binary`
  - `json`
  - `boolean`
  - `null`
  - `unknown`
- 执行 DQL。
- 从实际结果值或驱动返回类型中提取实际类型族。
- 对每一列比较推断类型与实际类型。

Bug 判定：

- 实际结果类型族与逻辑推断类型族不一致，并且该差异不是已知 metadata 暴露行为。

注意事项：

- `YEAR(...)` 的逻辑结果应视为 `numeric`。
- MySQL 可能在某些上下文中通过 metadata 暴露为 `YEAR/datetime`，这类现象应在对比层标记为 metadata behavior，不应改写核心规则。

可检测问题：

- 函数返回类型错误。
- 条件表达式隐式类型转换错误。
- set operation 公共类型合并错误。
- binary/string 混合时返回类型错误。

### 2.3 结果列可空性约束

推断内容：

- `NOT NULL` 基础列直接投影通常为非空。
- nullable 列直接投影为可空。
- `COUNT(*)` 一定非空。
- `COUNT(expr)` 一定非空。
- `SUM`、`AVG`、`MIN`、`MAX` 在空输入下可能为 NULL。
- `IFNULL(a, b)`、`COALESCE(...)` 可根据参数可空性推断。
- `LEFT JOIN` 右表列会被 null-extension，通常变为可空。
- `INNER JOIN` 不会额外引入 null-extension。

比较方案：

- 推断每个结果列是否允许为 NULL。
- 执行 DQL 并扫描结果。
- 如果某列被推断为非空，则实际结果中该列不能出现 NULL。

Bug 判定：

- 推断为非空的列实际返回了 NULL。

可检测问题：

- NOT NULL 约束失效。
- join null-extension 处理错误。
- 聚合函数空值处理错误。
- 表达式 null 传播错误。

### 2.4 结果行数上界约束

推断内容：

- `LIMIT n` 表示结果行数必须小于等于 n。
- 无 `GROUP BY` 的纯聚合查询通常最多返回 1 行。
- 标量子查询最多允许返回 1 行。
- `WHERE FALSE`、`LIMIT 0` 等可推出结果行数为 0。

比较方案：

- 推断结果行数上界。
- 执行 DQL。
- 比较 `actual_row_count <= inferred_upper_bound`。

Bug 判定：

- 实际行数超过逻辑上界。

可检测问题：

- LIMIT 失效。
- 聚合无分组场景返回多行。
- 标量子查询行数约束失效。
- 查询重写导致重复行异常。

### 2.5 DISTINCT / UNION 去重约束

推断内容：

- `SELECT DISTINCT ...` 的最终结果不允许出现重复行。
- `UNION` 默认去重。
- `UNION ALL` 不要求去重。
- 如果 set operation 外层还有 `DISTINCT`，最终结果也应去重。

比较方案：

- 判断 DQL 最终结果是否具有去重语义。
- 执行 DQL。
- 对实际结果按整行做规范化哈希。
- 检查是否存在重复行。

Bug 判定：

- 具有去重语义的结果中出现重复行。

可检测问题：

- DISTINCT 去重失效。
- UNION 去重失效。
- 排序去重或哈希去重错误。
- 临时表物理布局或比较逻辑错误。

### 2.6 ORDER BY 有序性约束

推断内容：

- 最终 `ORDER BY` 定义了结果行的排序约束。
- ASC 要求非递减。
- DESC 要求非递增。
- 多列排序按字典序比较。
- NULL 排序位置需要按具体 DBMS 方言建模。

比较方案：

- 推断最终排序 key 与排序方向。
- 执行 DQL。
- 根据实际结果中的排序表达式值检查相邻行是否满足顺序。

Bug 判定：

- 实际返回顺序违反 `ORDER BY` 约束。

可检测问题：

- 排序算子错误。
- 索引扫描顺序错误。
- `ORDER BY + LIMIT` 优化错误。
- NULL 排序处理错误。

注意事项：

- 如果 `ORDER BY` 表达式没有出现在 SELECT 结果中，需要额外执行包装查询或在生成阶段保证排序 key 可观测。

### 2.7 函数返回值域约束

推断内容：

部分函数的返回值具有稳定值域：

- `MONTH(x)` 返回 1 到 12 或 NULL。
- `DAYOFWEEK(x)` 返回 1 到 7 或 NULL。
- `DAYOFMONTH(x)` 返回 1 到 31 或 NULL。
- `DAYOFYEAR(x)` 返回 1 到 366 或 NULL。
- `HOUR(x)` 返回 0 到 23 或 NULL。
- `MINUTE(x)` 返回 0 到 59 或 NULL。
- `SECOND(x)` 返回 0 到 59 或 NULL。
- `LENGTH(x)`、`CHAR_LENGTH(x)` 返回非负整数或 NULL。
- `COUNT(*)` 返回非负整数且非 NULL。
- `ABS(x)` 返回非负数或 NULL。

比较方案：

- 识别结果列中的函数表达式。
- 推断该表达式的值域约束。
- 执行 DQL 并扫描对应结果列。
- 检查每个非 NULL 值是否落在值域内。

Bug 判定：

- 实际值超出函数逻辑值域。

可检测问题：

- 日期函数计算错误。
- 字符串长度函数错误。
- 聚合计数错误。
- 数值函数边界错误。

### 2.8 谓词约束传播

推断内容：

当 WHERE 谓词直接约束了输出列时，可以将谓词传播到结果：

- `SELECT c FROM t WHERE c IS NOT NULL` => 输出 `c` 不能为 NULL。
- `SELECT c FROM t WHERE c > 10` => 输出 `c` 必须大于 10。
- `SELECT c FROM t WHERE c BETWEEN 1 AND 5` => 输出 `c` 必须在 `[1, 5]`。
- `SELECT c FROM t WHERE c IN (1, 2, 3)` => 输出 `c` 必须属于集合。
- `SELECT c FROM t WHERE c = 5` => 输出 `c` 必须等于 5。

比较方案：

- 建立输出列与底层列的 lineage。
- 从 WHERE 中提取可传播谓词。
- 如果输出列直接依赖同一底层列，则将谓词转化为结果列约束。
- 执行 DQL 并扫描实际结果。

Bug 判定：

- 实际结果值违反 WHERE 谓词推出的约束。

可检测问题：

- 谓词过滤错误。
- 谓词下推错误。
- 索引条件下推错误。
- 类型转换比较错误。
- NULL 逻辑处理错误。

限制：

- 复杂表达式、隐式类型转换、OR 条件和非确定性函数需要谨慎处理。
- 初期只处理简单列引用与简单比较谓词。

### 2.9 聚合结果自洽约束

推断内容：

同一 SELECT 中多个聚合结果之间存在逻辑关系：

- `COUNT(col) <= COUNT(*)`。
- `MIN(x) <= MAX(x)`。
- 当 `COUNT(x) > 0` 时，数值列 `AVG(x)` 应位于 `MIN(x)` 和 `MAX(x)` 之间。
- `COUNT(*) >= 0`。
- `SUM(x)`、`AVG(x)`、`STDDEV(x)` 等结果类型应为 numeric。

比较方案：

- 识别同一查询块中的聚合表达式。
- 推断聚合间约束。
- 执行 DQL。
- 对每一行聚合结果检查自洽性。

Bug 判定：

- 聚合结果之间违反数学关系。

可检测问题：

- 聚合状态计算错误。
- GROUP BY 分组错误。
- NULL 聚合规则错误。
- 聚合临时表或索引读取错误。

### 2.10 GROUP BY 唯一性约束

推断内容：

- 如果结果中包含完整 group key，则每个 group key 在最终结果中最多出现一次。
- 无 `GROUP BY` 的聚合查询最多返回一行。

比较方案：

- 识别 GROUP BY key。
- 判断 SELECT 结果中是否包含完整 key。
- 执行 DQL。
- 对结果中的 group key 做去重检查。

Bug 判定：

- 同一 group key 在结果中出现多次。

可检测问题：

- 分组错误。
- group key 比较错误。
- hash aggregate 或 sort aggregate 错误。

## 3. 辅助推断能力

### 3.1 Lineage 推断

Lineage 本身通常不直接判定 bug，但它是其他约束的基础。

推断内容：

- 每个结果列依赖哪些底层表列。
- 表达式依赖集合。
- CTE、派生表、子查询的列映射。
- 聚合表达式依赖的输入列。

用途：

- 支持谓词约束传播。
- 支持 nullability 推断。
- 支持 metadata 变异对象选择。
- 支持结果列与底层 schema 的关联。

### 3.2 非确定性表达式识别

推断内容：

- `RAND()`、`UUID()`、`NOW()` 等非确定性函数。
- 依赖当前时间、当前连接状态、随机数的表达式。

用途：

- 避免对非确定性结果做严格等价比较。
- 对结果比较策略进行降级或跳过。

### 3.3 引用对象集合推断

推断内容：

- DQL 引用了哪些表、列、CTE、函数、索引候选列。

用途：

- 指导 metadata 变异只作用于 DQL 实际依赖对象。
- 减少无效变异。
- 提升 bug 定位能力。

## 4. 建议实现优先级

第一阶段：低成本、高确定性约束。

1. 结果列数量。
2. 结果列逻辑类型。
3. LIMIT 行数上界。
4. DISTINCT / UNION 去重性。

第二阶段：需要 lineage 支持的约束。

1. 结果列可空性。
2. 谓词约束传播。
3. GROUP BY 唯一性。

第三阶段：语义更强但实现复杂的约束。

1. ORDER BY 有序性。
2. 函数返回值域。
3. 聚合结果自洽性。
4. 非确定性表达式识别。

## 5. 推荐模块划分

```text
metadata_inference/
  column_count.py        # 结果列数量推断
  column_names.py        # 结果列名推断
  column_types.py        # 结果列逻辑类型推断
  nullability.py         # 结果列可空性推断
  lineage.py             # 结果列来源依赖推断
  result_shape.py        # LIMIT / DISTINCT / GROUP BY / ORDER BY / setop 形状推断
  value_constraints.py   # 函数值域、谓词传播、聚合自洽约束
  dependencies.py        # 表、列、函数、CTE 引用集合推断
```

## 6. 约束比较流程

标准流程：

1. 解析 DQL AST。
2. 绑定 schema 与作用域。
3. 推断结果列数量、类型、可空性、lineage。
4. 推断结果形状约束，例如 LIMIT、DISTINCT、ORDER BY、GROUP BY。
5. 推断值域约束与谓词传播约束。
6. 执行 DQL，获取实际结果。
7. 按约束逐项比较。
8. 如果违反约束，输出：
   - DQL 原文。
   - 推断约束。
   - 实际结果摘要。
   - 违反的约束类型。
   - 涉及的表、列、函数、CTE。

## 7. 不应放入核心逻辑推断的内容

以下内容不应作为核心逻辑推断规则：

- MySQL `cursor.description` 在不同上下文中的 metadata 暴露差异。
- 优化器执行计划。
- 实际扫描路径。
- 运行期错误，例如数值溢出、非法 JSON、非法日期。
- 完整 expected result。
- 谓词选择率。
- 精确结果行数，除非 SQL 逻辑能直接推出。

这些内容可以作为对比层、统计层或执行层信息保留，但不应反向修改核心逻辑推断规则。
